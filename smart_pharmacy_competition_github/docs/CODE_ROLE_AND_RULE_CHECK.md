# 代码作用与比赛规则核查

本文档用于说明本仓库各部分代码的作用，以及它们和智慧药房比赛流程之间的对应关系。

## 比赛流程抽象

1. 小车从起点出发。
2. 到达识别板 1，识别二维码任务。
3. 二维码给出取样窗口 A/B/C 与样本类型。
4. 小车到对应窗口取样并播报。
5. 小车经过识别板 2，读取化验区状态。
6. 若化验区空闲，应在规则要求时间内通过；若忙碌，应等待对应秒数后通过。
7. 小车到对应化验窗口 1/2/3/4 送样并播报。
8. 小车持续向裁判软件上传任务、速度、里程/位姿、视觉识别结果和状态。

## 代码模块对应关系

| 文件 | 作用 |
| --- | --- |
| `scripts/qrcode_voice_node.py` | 订阅相机图像，识别识别板 1 的二维码，发布 `/judge/cv1` 和任务状态，并触发语音播报。 |
| `scripts/board2_ocr_node.py` | 识别识别板 2 的文字状态，发布 `/judge/cv2`；可用于空闲/等待逻辑。 |
| `scripts/judge_bridge.py` | 小车端 TCP 客户端，连接电脑裁判软件 `192.168.12.248:8888`，按 JSON 行协议上传 `task/speed/odom/CV1/CV2`。 |
| `scripts/manual_arrival_voice_node.py` | 当到达 A/B/C 或 1/2/3/4 窗口时，根据识别板 1 的任务播报取样/送样内容。 |
| `scripts/auto_arrival_detector.py` | 根据 `map -> base_footprint` 实时位姿和 `waypoints_real.yaml`，自动判断是否到达窗口并触发播报。 |
| `scripts/patrol_mission.py` | 可选自主巡航任务节点，向 `move_base` 发送目标点；现场稳定性不足时不建议作为主模式。 |
| `launch/manual_competition.launch` | 推荐现场模式：只启动识别、语音、裁判同步和到点检测，不让程序直接控制速度。 |
| `pc_tools/keyboard_control_auto_login.py` | PC 侧 SSH 键盘控制工具，发布 `/cmd_vel`，比赛调试时作为唯一速度控制源。 |

## 当前推荐模式

当前最稳妥的比赛模式是：

```text
键盘控制小车运动
+ 识别板 1 二维码识别
+ 到点播报
+ 识别板 2 默认 FREE 或 OCR
+ 裁判系统 JSON 实时同步
```

这样可以避免 `move_base` 在定位跳点或局部规划失败时导致小车卡停，同时仍满足视觉识别、播报和裁判数据上传。

## 风险点

- 自主巡航是否能完整跑一圈，取决于定位、地图比例、代价地图膨胀和局部规划器稳定性。
- 自动到点检测依赖 `map -> base_footprint` TF；如果 RViz 中位姿明显漂移，应改用手动到点触发。
- 识别板 2 若现场没有贴内容，可以将 `INITIAL_CV2=FREE`，表示默认空闲。
- 公开 GitHub 时不要上传裁判软件 exe、真实 SSH 密码、日志和临时截图。

## 结论

本仓库已经具备比赛所需的核心链路：

```text
视觉识别 -> 任务解析 -> 语音播报 -> 裁判 JSON 上传 -> 键盘/可选自动控制
```

正式线下比赛建议优先使用手动驾驶保守模式；自主模式应只在现场连续验证稳定后启用。
